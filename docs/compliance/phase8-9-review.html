TaskBugStatusEvidence / Notes
T284Function call callee resolution PASS llvm.rs:430–433: Terminator::Call checks MirConstant::String(name) and emits @{name}. MIR builder uses function_map (mir.rs:941–948). Recursive and cross-function calls work — verified by e2e_fibonacci, e2e_mutual_recursion.
T285 String literal length not passed to print_str FAIL llvm.rs:1083–1091: emit_constant for MirConstant::String only returns a i8* GEP pointer. Length is never emitted as a second argument. T286 Rvalue::Aggregate type = UNIT FAIL llvm.rs:1167: type_of_rvalue still returns TypeId::UNIT for all Rvalue::Aggregate. Comment: "simplified". This causes struct/tuple assignments to be silently dropped in emit_assign (line 314: ty_str == "{}" guard). Struct literals are broken.
T287 emit_place_store ignores projections FAIL llvm.rs:1054–1064: For projected places, falls through to storing at base local. Comment: "Simplified: store to base local." Struct field writes and array element writes are no-ops.
T288 emit_place_load field type tracking PARTIAL llvm.rs:1001–1042: GEP chains ARE emitted for Field, Index, and Deref projections. However, after a Field projection, current_ty_str is hardcoded to "i8*" (line 1015), making subsequent loads use wrong types. Nested field access (a.b.c) will produce incorrect IR.
T289 Enum aggregate: tag-only, no payload FAIL llvm.rs:901–910: emit_aggregate for AggregateKind::Enum only inserts the tag byte. Comment: "Payload would require knowing the enum layout; simplified here." Enum variants with data are broken.
T290 Float constant format PASS llvm.rs:1070–1073: Correctly uses hex float format 0x{:016X}. E.g. 3.14 → 0x40091EB851EB851F. LLVM accepts this universally.
T291 Rvalue::Len always returns 0 FAIL llvm.rs:525–531: Still emits add i64 0, 0. Comment: "simplified: return 0". Array and string length are always zero.
T292 type_of_rvalue(Ref) returns INT64 FAIL llvm.rs:1166: Rvalue::Ref { .. } => TypeId::INT64. Comment: "simplified". References have wrong type in IR.
T293 print_bool ABI mismatch (i1 vs i8) PARTIAL IR declaration changed to i8 (runtime.rs:112), C function uses int8_t (runtime.rs:220) — both match. MIR builder registers "print" → "axon_print" in function_map (mir.rs:282) but axon_print is not a real runtime function. None of the 11 E2E tests call print(). 5 · Findings & Recommendations

🔴 Critical (3)

C-01: print() / println() builtin not implemented (T294) Files: src/typeck.rs, src/mir.rs, src/codegen/llvm.rs
The spec's #1 exit criterion is "./hello prints Hello, World!". Without a print() builtin, no Axon program can produce output. The type checker has zero recognition of print, and MIR maps it to the non-existent axon_print. This is the single most important missing feature.
Recommendation: Implement type-dispatched print(expr) in typeck → MIR → codegen as described in T294. Must handle Int64→axon_print_i64, Float64→axon_print_f64, Bool→axon_print_bool, String→axon_print_str(ptr, len). Add println() variant that appends axon_print_newline().
C-02: 7 of 12 LLVM IR codegen bugs remain unfixed (T285–T292) Files: src/codegen/llvm.rs
The spec identifies 12 specific correctness bugs. Only 5 are fixed (T284, T290, T293/partial, T295, T283). The remaining 7 affect: string length passing (T285), struct/tuple aggregate types (T286), projected stores (T287), enum payloads (T289), array/string length (T291), reference types (T292), and projected load types (T288/partial). These bugs block struct, enum, tuple, string, and reference features end-to-end.
Recommendation: Address in priority order: T285 (string length) → T286 (aggregate types) → T287 (projected stores) → T288 (projected load types) → T289 (enum payloads) → T291 (Len) → T292 (Ref type).
C-03: E2E tests verify exit code only, not output correctness (T300–T302) Files: tests/e2e_tests.rs, tests/e2e/*.axon
All 15 E2E tests only assert exit code == 0. No test checks stdout content against expected output. No // expect: header comments in .axon files. A program that computes wrong results but doesn't crash will pass. This creates false confidence — the spec requires stdout == expected assertion for every test.
Recommendation: Add // expect: <line> comments to all .axon files. Modify compile_and_run() to capture stdout. Add assert_output(source, expected_stdout) helper. Once print() works, add print calls to all test programs and verify output.

🟠 Major (4)

M-01: No cargo-fuzz integration (T251–T252, T254) Files: fuzz/
The spec requires libFuzzer-based targets. Only text descriptions and lightweight test-based fuzzing exist. No 48h campaigns, no crash corpus, no regression suite for CI. The fuzz/README.md explicitly acknowledges this as "Future."
Recommendation: Add fuzz/Cargo.toml, create fuzz/fuzz_targets/lexer_fuzz.rs etc. with fuzz_target! macro. Run at least one 48h campaign per target and save the corpus.
M-02: Missing main function validation — E5009/E5010 (T282) Files: src/codegen/llvm.rs or src/main.rs
Programs without fn main() fail with an opaque clang linker error instead of a helpful E5009 message. fn main(x: Int64) is not caught.
Recommendation: After type checking, scan TypedProgram.items for a function named "main". Emit E5009 if absent, E5010 if params ≠ 0 or return type ≠ Unit/Int64.
M-03: TensorHeader stub in C runtime (T296) Files: src/codegen/runtime.rs
Tensor allocation is a stub that ignores dtype sizes, has no TensorHeader struct, and shape_check always returns true. While tensor E2E is out of scope for Phase 9's scalar focus, this was explicitly specified in T296.
Recommendation: Implement the TensorHeader struct as specified. Proper allocation with element-size awareness. Shape comparison in axon_tensor_shape_check.
M-04: 4 of 15 required E2E test programs missing (T301) Files: tests/e2e/
Missing: structs.axon, enums.axon, tuples.axon, strings.axon (and others for unary ops, comparisons, type casts). These are blocked by unfixed codegen bugs (T286, T287, T289).
Recommendation: After fixing codegen bugs, add the missing test programs with // expect: annotations.

🔵 Minor (5)

m-01: Benchmarks don't use criterion.rs benches/compiler_bench.rs uses std::time::Instant in #[test] functions instead of criterion's statistical benchmarking. No JSON output, no regression detection.
Recommendation: Migrate to criterion.rs with criterion_group! and criterion_main! macros.
m-02: Runtime panic source location hardcoded (T305) Assert terminator passes "axon_module.axon" and line 0 instead of real file/line.
Recommendation: Add file: String and line: u32 to Terminator::Assert. Propagate from Span in MIR builder.
m-03: Missing tutorials 05, 06 and from-rust.md migration guide docs/tutorial/ has 4/6 tutorials. docs/migration/ missing from-rust.md.
Recommendation: Write 05-training-loop.md, 06-deployment.md, and from-rust.md.
m-04: --emit-obj doesn't include runtime linkage Uses old compile_ir_to_object() which only compiles IR. Resulting .o has unresolved runtime symbols.
Recommendation: Either also compile runtime.c to .o alongside, or document that the user must link axon_runtime.o manually.
m-05: No Miri, no sanitizers, no cargo audit in CI Spec requires Miri runs (T257), sanitizers (T258), and dependency auditing in CI (T259).
Recommendation: Add cargo +nightly miri test, cargo audit steps to CI. Consider ASan for compiled test programs.

9 · Verdict & Recommendation

⛔ NEEDS REVISION

Phase 8 (78%) is close to approval — documentation, release engineering, and compliance are excellent. The gaps are in tooling quality (no real fuzz targets, no Miri/sanitizers). These are low-risk for alpha release.

Phase 9 (57%) needs significant work. The build pipeline (compile_and_link, main wrapper, C runtime) is solid and E2E tests prove basic programs compile and run. However, the primary goal — ./hello printing output — cannot be achieved because print() is not implemented. 7 of 12 identified codegen bugs remain unfixed, blocking structs, enums, tuples, and strings. E2E tests provide false confidence by checking only exit codes.
