Phase 3 Findings

MAJOR F-002: No CFG-based borrow analysis

The spec requires building a control flow graph (T092) and computing variable liveness (T093) for borrow checking. The implementation uses a direct AST walk instead. This means borrow checking will not correctly handle branching control flow where a variable is moved in one branch but not another.

File: src/borrow.rs:83–90 — BasicBlock struct defined but never populated or traversed

Implement CFG construction from the TAST and use dataflow analysis for liveness. The existing walk can serve as a fallback for simple cases.

MAJOR F-003: No let-polymorphism / generalization

The spec requires Hindley-Milner generalization at let-bindings (Section 4.1: "Generalize — after checking a function body, generalize remaining free type variables into generic parameters"). The type checker unifies and substitutes but never generalizes, so polymorphic let-bindings won't work as expected.

File: src/typeck.rs — No generalize() function; apply_substitutions() resolves all type vars to concrete types

Add a generalization step after checking each function body to convert remaining free type variables into generic forall-quantified parameters.

MAJOR F-004: Simplified trait resolution

The spec requires a two-phase method resolution (inherent impls first, then trait impls) with coherence rules and orphan rule. The implementation uses a flat symbol table lookup with Type::Method naming convention, which won't scale to complex trait hierarchies or multiple trait implementations.

File: src/typeck.rs:851–904 — Method calls resolved via format!("{}::{}", type_name, method)

Implement a proper impl registry that separates inherent impls from trait impls, with explicit resolution order and coherence checking.

MINOR F-005: No match exhaustiveness checking

Pattern types are checked, but there is no warning or error when match arms don't cover all cases of an enum type. The spec mentions "match exhaustiveness basic" (T085).

File: src/typeck.rs — check_match() validates arm types but doesn't verify completeness

Implement basic exhaustiveness checking: at minimum, warn when a non-wildcard match on an enum is missing variants.

Phase 3 Verdict: ⚠️ APPROVED WITH CONDITIONS — Core type checking, shape verification, and error code coverage are excellent. The simplified borrow checker and trait resolution are functional for basic programs but will need hardening for complex real-world code.

Phase 4 Findings

CRITICAL F-006: LLVM backend uses textual IR instead of inkwell/llvm-sys

The spec requires programmatic LLVM IR construction via inkwell or llvm-sys (T114). The implementation generates textual .ll files by concatenating strings, then invokes clang to compile them. This approach:

  • Cannot run LLVM optimization passes directly (Mem2Reg, GVN, vectorization)
  • Cannot emit DWARF debug info via DIBuilder
  • Is fragile — string-based IR generation can produce invalid IR silently
  • Has no LLVM verification pass to validate generated IR

File: src/codegen/llvm.rs:26–28 — LlvmCodegen uses output: String; Cargo.toml has no LLVM dependencies

This is the highest-priority architectural debt. Options: (1) Add inkwell dependency and rewrite codegen to use LLVM builder APIs, or (2) formalize this as a "textual backend" and document the trade-offs.

CRITICAL F-007: MLIR backend is a non-functional stub

The spec requires GPU code generation through MLIR Linalg→MemRef→GPU→PTX/SPIR-V (T122–T126). The implementation is a 90-line file with a compile_gpu() that returns "not yet implemented". Tasks T123–T126 were redefined in tasks.md to documentation-only tasks ("GPU function detection", "Architecture documentation") rather than actual MLIR implementation.

File: src/codegen/mlir.rs:47–54 — compile_gpu() returns immediate error

Acknowledged as a deliberate deferral. Recommend documenting this clearly in README as "GPU backend: planned, not yet implemented" and targeting for Phase 8+.

MAJOR F-008: No DWARF debug info emission

T120 requires DWARF debug info for source-level debugging with GDB/LLDB. Zero debug metadata is emitted. This blocks the "Debug builds produce DWARF info" acceptance criterion from the Phase 4 spec §9.

File: src/codegen/llvm.rs — no !dbg, no !DISubprogram, no !DICompileUnit

Add LLVM metadata nodes (!DIFile, !DISubprogram, !DILocation) to the textual IR output. Textual IR does support debug metadata strings.

MAJOR F-009: Missing refcount runtime functions

The spec (§5.1) requires axon_refcount_inc(ptr) and axon_refcount_dec(ptr) → bool for Arc<T> support. These are absent from both the LLVM declarations and the C runtime source.

File: src/codegen/runtime.rs — RUNTIME_FUNCTIONS has 15 entries; no refcount functions

Add axon_refcount_inc/axon_refcount_dec to runtime declarations and C source.

MAJOR F-010: Missing AddressOf Rvalue variant

The MIR spec (§2.2) defines Rvalue::AddressOf { place } as a distinct operation from Ref. The implementation has Ref but no AddressOf. While functionally similar, the distinction matters for unsafe raw pointer semantics.

File: src/mir.rs:147–181 — Rvalue enum has 9 variants; no AddressOf

Add AddressOf { place } to Rvalue and emit it for raw pointer operations. Low priority until unsafe pointers are needed.

MINOR F-011: Optimization delegated to clang, not internal LLVM passes

The spec §3.3 describes specific LLVM optimization passes per level (Mem2Reg, SROA, GVN, etc.). The implementation passes -O0 through -O3 flags to clang, which applies its own pass pipeline. Functionally equivalent but less control over which passes run.

File: src/codegen/llvm.rs:1264–1270 — opt_flag forwarded to Command::new("clang")

Document this as "clang-delegated optimization" rather than "internal LLVM pipeline". Consider LLVM-aware optimization if moving to inkwell.

MINOR F-012: Projected place store is incomplete

When storing to a projected place (e.g., struct field assignment s.x = 5), the LLVM codegen falls back to storing to the base local, ignoring projections. This means field assignment doesn't work correctly in generated LLVM IR.

File: src/codegen/llvm.rs:1054–1064 — comment says "Simplified: store to base local"

Implement GEP chain for projected stores, mirroring the projected load implementation.

MINOR F-013: Tensor op codegen returns placeholder values

The LLVM codegen for tensor operations (matmul, add, etc.) returns the left operand as a placeholder rather than calling runtime functions. For example, emit_tensor_op(MatMul) emits bitcasts but returns lhs unchanged.

File: src/codegen/llvm.rs:923–963 — "Placeholder: return lhs" comments

Implement actual runtime calls (axon_tensor_matmul, etc.) when tensor runtime functions exist.

MINOR F-014: MIR Display has extra parenthesis typo

The StorageLive and StorageDead Display implementations emit an extra closing parenthesis: "StorageLive(_{}))" instead of "StorageLive(_{})".

File: src/mir.rs:1516–1519

Remove the extra ) in the format strings.

Phase 4 Verdict: ❌ NEEDS REVISION — MIR lowering is excellent (fully spec-compliant). LLVM backend is functionally operational but architecturally non-compliant (textual IR vs. inkwell). MLIR backend is a non-functional stub. Critical missing features: debug info, refcounting, projected stores.

Anti-Patterns & Concerns

  • Placeholder returns in codegen — Multiple points where codegen returns "undef" or the first operand instead of actual computation (tensor ops, aggregates, projected places)
  • Task redefinition — Tasks T122–T126 were redefined in tasks.md from their spec definitions (e.g., "bufferization pass" → "GPU function detection"). This masks missing features.
  • Unused dead code — BasicBlock in borrow.rs, #[allow(dead_code)] on opt_level field in LlvmCodegen
  • Incomplete Display impls — StorageLive/StorageDead have extra parenthesis; minor but indicates lack of output testing
  • Demangle is a placeholder — NameMangler::demangle() returns the raw symbol, not a demangled form

7. Top Recommendations

Priority 1: Formalize the LLVM textual IR approach (F-006)

Either (a) add inkwell and rebuild the codegen, or (b) formally accept the textual approach and add an LLVM IR validator step (llvm-as → /dev/null) to catch malformed IR early. Add DWARF metadata as textual annotations.

Priority 2: Fix projected place stores (F-012)

Struct field assignment (s.x = 5) is broken in codegen. Implement GEP chains for projected stores to match the load implementation. This blocks real programs.

Priority 3: Implement CFG-based borrow checking (F-002)

The MIR already has a proper CFG. Reuse MIR's BasicBlock/Terminator structure for borrow analysis instead of the current AST walk. This is needed for correctness on branching control flow.
