Bend Language Analysis

A comprehensive analysis of Bend's innovations and how they can be adopted to improve the Axon programming language.

Executive Summary

Bend is a massively parallel programming language by HigherOrderCO that compiles to interaction combinator nets (HVM2), enabling automatic parallelism across CPUs and NVIDIA GPUs without threads, locks, or annotations. While Axon and Bend target different niches — Axon is an ML-first systems language with compile-time tensor shapes, while Bend is a pure-functional parallel computing language — Bend introduces several innovative ideas that Axon can adopt to significantly strengthen its compiler, runtime, parallelism model, and developer experience.

1. Implicit Parallelism Model

Critical

Adopt data-dependency-based auto-parallelism for tensor operations

Bend's killer feature is implicit parallelism: no threads, no locks, no annotations. Independent computations automatically execute in parallel. While Axon shouldn't adopt interaction combinators (they sacrifice single-core performance), the principle is directly applicable to tensor workloads.

What Bend Does

  • Programs compile to interaction nets where independent redexes reduce in parallel
  • H(f(x), g(y)) runs f(x) and g(y) in parallel automatically
  • Same source code, different backends: sequential (Rust), parallel (C/pthreads), massively parallel (CUDA)
  • Tree-structured computations scale to thousands of threads

How Axon Should Adapt This

  • Build a tensor computation graph at compile time — analyze data dependencies between tensor operations in a function and identify independent subgraphs that can execute in parallel
  • Auto-parallelize independent tensor ops — e.g., if val a = matmul(x, w1) and val b = matmul(x, w2) appear in the same scope, schedule them concurrently on GPU without user annotation
  • Multi-backend from same source — axonc build --target cpu vs axonc build --target cuda should produce semantically identical programs with different parallelism strategies
  • MIR-level dependency analysis — already have CFG-based MIR; add a data-dependency graph pass that identifies parallelizable regions
Implementation path: Add a src/mir/parallelize.rs pass that builds a DAG of tensor operations, identifies independent subgraphs, and emits parallel dispatch nodes in the MIR. For CPU: thread pool. For GPU: CUDA streams.

2. Fold & Bend Constructs

Critical

Add fold and reduce as first-class tensor operations

Bend's fold (consume recursive data) and bend (generate recursive data) are its primary loop replacements. While Axon has traditional loops, these constructs map perfectly to tensor reductions and parallel generation patterns that are central to ML workloads.

What Bend Does

// fold: consume a tree by recursing on ~-marked fields
fold tree:
  case Tree/Node:
    return tree.left + tree.right
  case Tree/Leaf:
    return tree.value

// bend: generate a tree in parallel
bend depth = 0:
  when depth < 10:
    tree = ![fork(depth + 1), fork(depth + 1)]
  else:
    tree = !1

How Axon Should Adapt This

// Tensor fold (parallel reduction)
val total = tensor.fold(0.0, |acc, x| acc + x)

// Batch-parallel map-reduce
val means = batches.fold(Tensor.zeros([10]),
  |acc, batch| acc + model.forward(batch)) / n

// Generate training data in parallel (bend-like)
val dataset = Tensor.generate([1000, 784], |i, j| {
  randn() * 0.1 + class_centers[i % 10][j]
})
Key insight: Bend's fold/bend are desugared into recursive functions that parallelize over tree structure. For Axon, fold over tensors should desugar into parallel reduction kernels, and generate should desugar into parallel initialization kernels. This gives Axon the same parallelism benefits without requiring immutable-only semantics.

3. Dual Syntax System

Medium

Consider a math-notation mode for tensor expressions

Bend lets you write the same function in Python-like or Haskell-like syntax, mixed freely in the same file. While Axon shouldn't add a full second syntax, a math-notation mode for tensor expressions would greatly improve readability for ML researchers.

What Bend Does

# Imperative (Python-like)
def add(x, y):
  return x + y

# Functional (Haskell-like)
add x y = (+ x y)

# Mixed freely in same file!

Proposed Axon Adaptation

// Standard Axon
val loss = cross_entropy(softmax(linear(x, w) + b), labels)

// Math-notation block (proposed)
@math {
  loss = CE(softmax(Wx + b), y)
  grad_w = d(loss)/d(w)    // symbolic differentiation syntax
  w' = w - lr * grad_w
}
Caution: Bend's dual syntax adds complexity to their parser (two full parsers + conversion layer). For Axon, a lightweight math-notation block is lower risk than a full dual syntax. Consider this for v2.0+ only.

4. Accumulative Diagnostics

Critical

Adopt Bend's accumulative error collection with configurable severity

Bend collects ALL errors and warnings throughout compilation instead of failing on the first error. Users see every problem at once, with per-warning severity control via CLI flags. This is a massive developer experience improvement.

What Bend Does

  • Diagnostics struct accumulates errors/warnings across all phases
  • fatal() checkpoints — only bail if errors have accumulated
  • 8 warning categories, each configurable: -W unused_definition, -D recursion_cycle, -A irrefutable_match
  • ANSI-colored grouped output (warnings yellow, errors red, grouped by file/origin)
  • Errors carry source spans with file/line/char locations

How Axon Should Implement This

  • Replace early-exit error handling in typeck/borrow/codegen with an accumulative Diagnostics collector
  • Add --warn, --deny, --allow CLI flags for per-category severity control
  • Group errors by file, phase (parse/type/borrow/codegen), then display all at once
  • Add fatal() checkpoints at phase boundaries (e.g., after parsing, after type checking)
Implementation: Create src/diagnostics.rs with a Diagnostics struct containing a BTreeMap<Phase, Vec<Diagnostic>>. Thread it through all compiler passes. Existing error codes (E0100, E2001, etc.) map naturally to categories.

5. Transform Pass Architecture

High

Adopt Bend's ordered-pass desugaring pipeline

Bend has 31+ ordered transformation passes that progressively simplify the AST. Each pass is a small, focused module with clear input/output contracts. This architecture makes the compiler much more maintainable and extensible than monolithic transformation phases.

What Bend Does

  • 31 files in src/fun/transform/, each a single-purpose pass
  • Clear ordering: check_shared_names → set_entrypoint → encode_adts → … → unique_names
  • Each pass takes a &mut Book and transforms it in place
  • Dead code elimination, constant folding, linearization, etc. are all separate passes

How Axon Should Adapt This

  • Axon's current pipeline (Lexer → Parser → TypeChecker → TAST → MIR → LLVM) is good but monolithic within phases
  • Break up the MIR phase into ordered passes: build_cfg → dead_code_elim → constant_fold → tensor_fusion → parallelize → optimize_memory
  • Each pass in its own file under src/mir/transform/
  • Add a PassManager that runs passes in configurable order with diagnostics between each
Benefit: This directly enables Bend-inspired optimizations (combinator floating, linearization, auto-parallelization) to be added as individual passes without touching existing code.

6. ADT Encoding Strategies

High

Learn from Bend's Scott/NumScott encoding for GPU-friendly enums

Bend offers two ADT encodings: Scott (pure lambda) and NumScott (numeric tag dispatch). NumScott is more efficient on the HVM. Similarly, Axon's enum representation should be optimized for GPU execution.

What Bend Does

// Scott encoding: Ctr(a,b) -> λa. λb. λC1. λC2. (C_i a b)
// NumScott encoding: Ctr(a,b) -> λa. λb. λx. (x TAG a b)
// NumScott uses integer tag dispatch instead of lambda selection
// Configurable via -Oadt-scott / -Oadt-num-scott

How Axon Should Adapt This

  • Current Axon enums use a discriminant + payload layout (standard Rust-like approach)
  • For GPU-targeted code, consider a tagged-union with integer dispatch (similar to NumScott) that avoids pointer-heavy layouts
  • Add a codegen option: --enum-layout packed|tagged|scattered for different target architectures
  • Particularly important for Option<Tensor> and Result<Tensor, E> in hot paths
Key insight: Bend's configurable encoding proves that different ADT representations suit different backends. Axon should offer at least two: one optimized for CPU (cache-friendly tagged union) and one for GPU (flat numeric dispatch).

7. Linearity & Affine Types

High

Add linear/affine type annotations for zero-copy tensor operations

Bend enforces that every variable is used exactly once (affine). Multi-use triggers automatic duplication. For Axon's tensor operations, explicit linearity annotations would enable guaranteed zero-copy operations and eliminate unnecessary tensor clones.

What Bend Does

  • Every variable used at most once; compiler inserts dup nodes for multi-use
  • linearize_vars pass: 0 uses → erase, 1 use → keep, N uses → insert dup pattern
  • The programmer can see where copies happen by looking for multi-use variables

How Axon Should Adapt This

// Proposed: linear tensor annotation
fn train_step(linear input: Tensor<F32, [B, 784]>): Tensor<F32, [B, 10]> {
    // `input` MUST be used exactly once — guaranteed zero-copy
    val h = relu(linear1.forward(input))  // input consumed here
    // Using `input` again would be a compile error
    linear2.forward(h)
}

// In-place mutation for unique tensors
fn update_weights(unique w: &mut Tensor<F32, [N, M]>, grad: &Tensor<F32, [N, M]>) {
    w -= lr * grad  // guaranteed in-place, no allocation
}
Benefit: This is critical for GPU performance. Tensor copies are the #1 hidden cost in ML training loops. Linearity annotations let the compiler guarantee zero-copy forwarding and in-place updates, directly translating to faster training.

8. Monadic IO Blocks

Medium

Adopt with/ask blocks for effectful tensor pipelines

Bend's with IO: blocks desugar ask (<-) into monadic bind, providing clean sequential syntax for effectful operations. Axon could use this pattern for training loops and data pipelines.

What Bend Does

def main() -> IO(u24):
  with IO:
    * <- IO/print("Hello!\n")
    line <- IO/input()
    return wrap(0)
// Desugars to: IO/bind(IO/print("Hello!\n"), λ*. IO/bind(IO/input(), λline. IO/wrap(0)))

Proposed Axon Adaptation

// Training pipeline with monadic data loading
fn train(loader: DataLoader) {
    with loader:
        val (images, labels) <- next_batch()
        val loss = model.forward(images) |> cross_entropy(labels)
        loss.backward()
        optimizer.step()
}

// GPU computation pipeline
fn inference(model: Model, input: Tensor) {
    with Device.GPU(0):
        val x <- transfer(input)    // async GPU transfer
        val y <- model.forward(x)   // GPU computation
        transfer_back(y)             // result back to CPU
}
Consideration: This overlaps with Axon's existing for..in loops for data loading. The with block adds value primarily for async/effectful pipelines (GPU transfers, distributed training). Implement after GPU codegen is working.

9. Snapshot/Golden Testing

High

Add snapshot testing for compiler output stability

Bend uses cargo-insta with 508 .bend test files and 452 snapshots to verify compiler output across all phases. This catches regressions in parsing, desugaring, codegen, and readback automatically. Axon should adopt this for LLVM IR output, MIR dumps, and error messages.

What Bend Does

  • 22 test categories: compile_file, run_file, parse_file, desugar_file, etc.
  • Each .bend file is a test input; corresponding .snap file is the expected output
  • cargo insta review for interactive snapshot approval
  • Tests cover: parsing output, desugared AST, compiled HVM nets, execution results, error messages

How Axon Should Implement This

  • Add insta as a dev dependency
  • Create tests/snapshots/ with categories: parse/, typecheck/, mir/, llvm_ir/, errors/
  • Each .axon file produces a snapshot of the relevant compiler output
  • Error message snapshots are particularly valuable — ensures error quality doesn't regress
Quick win: Start with error message snapshots. These are the most user-facing output and the most likely to regress during refactoring. Add MIR/LLVM IR snapshots later for codegen stability.

10. Combinator Floating & Inlining

High

Adopt closed-subexpression extraction for tensor kernel optimization

Bend's float_combinators pass extracts closed sub-terms (no free variables) to top-level definitions, reducing duplication and enabling better optimization. For Axon, this principle maps directly to tensor kernel extraction.

What Bend Does

  • Identifies closed sub-expressions (lambda bodies with no free variables)
  • Hoists them to top-level as {def}__C{N} combinators
  • Reduces interaction net size (critical for CUDA's 64-node limit)
  • Combined with inlining of trivial definitions for balance

How Axon Should Adapt This

  • Tensor kernel extraction: Identify sequences of tensor ops with no side effects → extract as standalone GPU kernels
  • Operator fusion: relu(matmul(x, w) + b) should fuse into a single fused_linear_relu kernel
  • MIR-level pass: Walk the MIR CFG, identify closed tensor operation subgraphs, extract to separate functions marked for GPU compilation
This is the GPU performance key. Modern ML compilers (XLA, TVM, Triton) all do operator fusion. Bend's combinator floating is the same principle applied to lambda calculus. Axon should do it at the MIR level for tensor ops.

11. Global String Interning

Medium

Enhance Axon's interner with Bend's global pool pattern

Bend uses a global STRINGS: GlobalPool<String> where all Name values are interned references. This makes name comparison O(1) and reduces memory usage. Axon already has a TypeInterner but should extend interning to all identifiers.

Current Axon State

  • TypeInterner in src/types.rs interns types for O(1) type equality
  • Variable names, function names, module paths are still heap-allocated String

Recommended Enhancement

  • Add a global NameInterner that all identifier strings pass through
  • Replace String identifiers in AST/TAST/MIR with interned Name handles
  • Benefits: faster comparison in type checker and borrow checker, less memory pressure, cache-friendlier AST traversal
Estimated impact: 10-20% reduction in compiler memory usage on large programs, measurable speedup in type checking due to O(1) name comparison.

12. List Comprehensions

Medium

Add tensor comprehension syntax

Bend has [x * 2 for x in list] with optional filters. Axon should extend this concept to tensor creation, which would feel natural to Python/NumPy users.

What Bend Does

[x * 2 for x in my_list]
[x for x in list if x > 0]
// Desugars to fold + conditional cons

Proposed Axon Adaptation

// Tensor comprehension
val weights = [randn() * 0.01 for _ in 0..784 for _ in 0..256]
    .reshape([784, 256])

// Filtered comprehension for sparse tensors
val mask = [1.0 if prob > 0.5 else 0.0 for prob in predictions]

// Matrix comprehension
val identity = [[if i == j { 1.0 } else { 0.0 } for j in 0..n] for i in 0..n]
Python compatibility win: ML researchers expect comprehension syntax from Python. This is syntactic sugar that desugars to Vec::from_iter + tensor construction. Low implementation cost, high ergonomic value.

13. Superpositions for GPU Branching

Exploratory

Study Bend's superposition model for GPU warp divergence optimization

Bend's {a, b} superpositions represent values that exist in multiple states simultaneously. While this is an interaction-net concept, the idea maps to GPU warp divergence optimization: computing both branches of a conditional and selecting the result, avoiding thread divergence.

Potential Axon Application

// Instead of branching (causes warp divergence on GPU):
@gpu fn activate(x: Float32): Float32 {
    if x > 0.0 { x } else { 0.0 }  // branch divergence!
}

// Compiler could automatically transform to:
@gpu fn activate(x: Float32): Float32 {
    x * (x > 0.0) as Float32  // branchless, no divergence
}
Long-term research: This is an advanced optimization. Study Bend's superposition semantics for inspiration, but implement as a standard LLVM-level branchless optimization pass for GPU-targeted code.

14. Stack-Safe Recursion

Low

Add stack growth protection for deep recursive types

Bend uses the stacker crate with maybe_grow(32KB, 1MB) to prevent stack overflow in deeply recursive AST traversals. Axon should adopt this for its own recursive descent parser and tree-walking passes.

What Bend Does

  • stacker::maybe_grow(32 * 1024, 1024 * 1024, || { ... }) on every recursive call in Clone/Drop/traversal
  • Transparent to the programmer — no API change
  • Prevents crashes on deeply nested programs

How Axon Should Implement

  • Add stacker = "0.1" to Cargo.toml
  • Wrap recursive descent parser calls and AST traversals with maybe_grow
  • Particularly important in: parser.rs (recursive descent), typeck.rs (recursive unification), mir.rs (CFG traversal)
Quick win: ~30 minutes to implement. Prevents rare but catastrophic crashes on deeply nested input.

15. Fusing Functions

Critical

Implement operation fusion for tensor computation graphs

Bend has "fusing functions" where compositions like f(f(f(x))) execute in O(1) instead of O(n) through interaction net fusion. The same principle — operator fusion — is the #1 optimization in modern ML compilers (XLA, TVM, Triton, TensorRT).

What Bend Does

// "Fusing addition" — 2^N additions in O(N) steps via interaction net fusion
Nat.add = @a @b
  a (@pred (Nat.add pred b))    // Succ case: recurse
  b                              // Zero case: return b

Critical Axon Implementation

// Axon should fuse these at compile time:
val y = relu(matmul(x, w) + b)
// Into a single kernel: fused_linear_relu(x, w, b)

// Element-wise chain fusion:
val z = exp(x + y) * sigmoid(x - y)
// Into: fused_elementwise(x, y) — one memory pass instead of five

// Reduction fusion:
val loss = mean(square(predictions - labels))
// Into: fused_mse(predictions, labels) — no intermediate tensors
This is the highest-impact optimization for ML performance. Every modern ML compiler does this. Without operator fusion, Axon will be 5-10x slower than PyTorch/JAX on real workloads due to memory bandwidth bottleneck from materializing intermediate tensors. Implement as a MIR pass that pattern-matches common operation sequences and replaces them with fused runtime calls.

Bend offers fine-grained compiler flags: -Oeta, -Oprune, -Omerge, -Oinline, -Ofloat-combinators, -Olinearize-matches, etc. This is invaluable for debugging compiler issues and benchmarking individual optimizations.

What NOT to Adopt from Bend

Not every Bend feature is appropriate for Axon. Here's what to explicitly avoid:

Bend FeatureWhy NOT for Axon
Interaction Combinators Axon targets LLVM native code. IC-based evaluation sacrifices single-core performance (Bend is 50-100x slower than C on single core). Keep LLVM IR.
24-bit Numbers Axon has full 8/16/32/64-bit numeric types. 24-bit is a severe limitation imposed by HVM's architecture. Axon has the right approach.
Pure Immutability Axon's val/var with ownership is strictly better for ML. In-place tensor updates are essential for training performance. Bend's immutability forces unnecessary copies.
No Loops Axon's for/while loops are essential for training loops, data loading, and imperative algorithms. Bend's fold/bend-only model is hostile to ML workflows.
Scopeless Lambdas ($x) These break referential transparency and make reasoning about code harder. Axon's closure model is cleaner.
Lambda-Encoded ADTs Scott/NumScott encoding makes pattern matching slower. Axon should use native discriminated unions with direct tag checks. Study encoding strategies for GPU only.
Full Dual Syntax Two complete parsers doubles maintenance cost. Bend's team maintains this, but for Axon it's not worth the complexity. A limited math-notation block is sufficient.
HVM Runtime Dependency Bend is tightly coupled to HVM (exact version pin). Axon should maintain its independence and compile to standard LLVM IR + native code.
